# ToLiveLong Overview

## 1) 문서 목적
- 제품의 `최종 목표 기능`과 `현재 구현 상태`를 분리해 관리한다.
- 기능 변경 시 어떤 문서를 어떻게 갱신할지 기준을 제공한다.
- 기준일: 2026-02-24

## 2) 최종 목표 기능 (Target)

### 2.1 공통 구조
- 하단 네비게이션: 대시보드(오늘), 기록/히스토리, 내 정보(My)
- 모바일 우선 UI

### 2.2 사용자 정보/목표
- 사용자 신체 정보 입력: 성별, 키, 몸무게, 출생연도, 활동량
- 목표 설정: 감량/유지/증량
- BMR/TDEE 기반 일일 권장 목표 자동 계산
- 목표값을 대시보드와 기록 화면 전반에 반영

### 2.3 식단 입력
- 수기 입력: 식사 타입, 음식명, 영양정보
- 음식 검색 입력: DB 기반 검색, 섭취량 기반 자동 계산
- 사진 입력: 음식/영양성분표 이미지 분석 후 자동 입력 + 사용자 확인/수정

### 2.4 대시보드(오늘)
- 총 섭취 칼로리/목표 칼로리 표시
- 탄/단/지/당/나트륨 요약 테이블
- 식사 타입별 로그 테이블
- 일간/주간 전환 관점 제공(요약 기준 일관성 유지)

### 2.5 히스토리
- 일/주 단위 필터
- 날짜별 영양 섭취 기록 목록
- 구간별 추세 확인(필요 시 차트 확장)

### 2.6 내 정보(My)
- 신체 정보 조회/수정
- 목표 조회/수정
- 수정 즉시 목표값 재계산 및 반영

### 2.7 운영/품질
- DB 스키마/코드/문서 정합성 유지
- 에러 처리와 재시도 UX 제공
- 테스트/검증 루틴 유지

## 3) 현재 구현 기능 (As-Is)

### 3.1 구현 완료
- 온보딩 입력 + 목표 계산(BMR/TDEE) + localStorage 저장
- 오늘 화면:
  - Supabase `daily_logs` 당일 조회
  - 칼로리 게이지 표시
  - 영양 요약 테이블 표시
  - 식사 타입별 테이블 표시
- 음식 검색 모달:
  - 로컬 음식 목록 검색
  - 즐겨찾기(localStorage)
  - 섭취량 비율 기반 영양값 계산 후 저장
- 사진 분석 모달:
  - `/api/analyze` 호출
  - Gemini 분석 결과를 확인/수정 후 저장
- 하단 네비게이션 기본 구조

### 3.2 미구현/부분 구현
- `history` 페이지: placeholder 상태
- `my` 페이지: placeholder 상태
- 음식 검색: DB 기반이 아닌 로컬 샘플 데이터 기반
- 스키마 정합성: 코드에서 `sugar`, `sodium` 사용하나 SQL 스키마에는 누락
- 인코딩 이슈: 일부 파일 한글 깨짐 이력 존재

## 4) 갭 요약 (Target vs As-Is)
- 히스토리/내정보 핵심 기능 미완성
- 데이터 소스 로컬 하드코딩 -> DB 기반 전환 필요
- 스키마/코드 불일치 해소 필요
- 문서/코드 동시 업데이트 프로세스 강화 필요

## 5) 기능 추가 시 문서 반영 플로우 (지침)

### 5.1 기본 원칙
- 기능 개발 PR은 코드 변경과 문서 변경을 반드시 함께 포함한다.
- 문서 반영 없는 기능 변경은 완료로 간주하지 않는다.

### 5.2 문서 연결 규칙
- 기능 상태 변경(구현/미구현 전환): `docs/overview.md` 갱신
- 아키텍처/스택/규칙 변경: `docs/Context.md` 갱신
- 해야 할 작업 추가/완료: `docs/Todo.md` 갱신
- 버그/실수/회귀 발생: `docs/MistakeNote.md` 기록
- 스키마 변경: `docs/Schema.md` 및 실제 SQL 파일 동시 갱신

### 5.3 작업 순서
1. 작업 시작 전 `docs/overview.md`, `docs/Context.md`, `docs/Todo.md` 확인
2. 구현 중 범위 변경 시 `docs/Todo.md` 먼저 수정
3. 구현 완료 후 `docs/overview.md`의 As-Is 상태 즉시 갱신
4. 구조/정책이 바뀌면 `docs/Context.md` 동시 갱신
5. 실수/장애가 있었다면 `docs/MistakeNote.md`에 원인과 재발방지 기록

### 5.4 PR 체크리스트
- [ ] 기능 코드 반영
- [ ] `docs/overview.md` 반영
- [ ] 필요 시 `docs/Context.md` 반영
- [ ] `docs/Todo.md` 상태 업데이트
- [ ] 이슈 발생 시 `docs/MistakeNote.md` 기록

## 6) 관련 문서
- 컨텍스트/구조/규칙: `docs/Context.md`
- 작업 백로그: `docs/Todo.md`
- 실수/회귀 기록: `docs/MistakeNote.md`
- 스키마: `docs/Schema.md`
